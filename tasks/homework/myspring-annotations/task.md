# Домашняя работа: @Component и @Autowired в MySpring

**Тег коммита:** `myspring-annotations`
**Баллы:** 7

---

## Описание

Расширьте MySpring: аннотации **своего** `@Component` и `@Autowired` (не из Spring) — поиск классов в пакете, регистрация бинов, внедрение зависимостей в поле или через конструктор.

Контекст поднимается **без JSON**: вызов **`ContextHolder.autoConfigure()`** (без параметров). Поведение `getBean(String id)` из практики сохраняете; дополнительно — **`getBean(Class<T> type)`**.

## Требования

- `@Component`: `RUNTIME`, на класс — кандидат в контейнер при сканировании.
- `@Autowired`: `RUNTIME`, на поле и на конструктор — точка внедрения.
- `ContextHolder.autoConfigure()` возвращает **контекст приложения** с `getBean(String id)` и `getBean(Class<T> type)`.
- Какой **корневой пакет** сканируется (и как компоненту сопоставляется `id` для `getBean(String)`, если используете строковый доступ), задайте в реализации и кратко опишите в коде или README к модулю — внешне для демо достаточно вызова без аргументов.
- Сначала создаются все компоненты, затем выполняется инъекция зависимостей.
- Внедрение через помеченное поле (в т.ч. `private`) и через помеченный конструктор; если конструктора с `@Autowired` нет — no-arg.
- Неоднозначность типа или отсутствие зависимости — `RuntimeException` с понятным текстом.
- Только стандартная библиотека Java (reflection, I/O, `java.net` при необходимости). Сторонние IoC и библиотеки сканирования classpath (Reflections, ClassGraph и т.п.) не использовать. Аннотации объявляете вы сами.

## Проверка

Сценарий тестов и демо:

```java
var context = ContextHolder.autoConfigure();
```

Далее — `context.getBean(…)` по типу и/или по `id`. Нужно покрыть три случая: компонент без зависимостей; с `@Autowired` полем; с `@Autowired` конструктором. Файл `context.json` для этого сценария не используется.
