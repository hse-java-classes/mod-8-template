# Практика: JSON IoC-контейнер MySpring

**Тег коммита:** `myspring-ioc`
**Баллы:** 5

---

## Описание

Реализуйте упрощённый IoC-контекст **MySpring**: чтение бинов из JSON-файла, создание экземпляров через reflection, связка через **конструкторы** (в т.ч. с зависимостями на другие бины и **вложенными** дескрипторами в JSON). Публичная точка входа — **`ContextHolder.getApplicationContext(path)`**.

Все бины из массива `beans` с полем `id` доступны по `id`. Запрос неизвестного `id` не бросает исключение — возвращается `null`.

## Структура JSON

Корень — объект с полем **`beans`**: массив дескрипторов верхнего уровня.

### Дескриптор бина (элемент `beans`)

| Поле | Обязательность | Смысл |
|------|----------------|--------|
| `id` | да, для элементов в `beans` | Идентификатор в контексте |
| `class` | да | Полное имя класса |
| `constructorArgs` | нет | Массив аргументов конструктора **в порядке параметров**. Если поля нет или массив пуст — используется **no-arg** конструктор |

### Элемент массива `constructorArgs`

Каждый элемент — один из вариантов:

1. **Ссылка на бин:** объект вида `{ "ref": "<id>" }` — в конструктор передаётся уже созданный экземпляр бина с данным `id` из контекста (в том числе объявленный выше в том же файле).
2. **Вложенный бин:** объект с полем **`class`** и опционально **`constructorArgs`** (та же рекурсивная схема). Такой объект описывает отдельный экземпляр, создаваемый при сборке родителя; у него **нет** `id` в контексте, если вы специально не договорились иначе в реализации. Вложенность может быть любой глубины.

Пример (цепочка зависимостей и вложенный бин в аргументе конструктора):

```json
{
  "beans": [
    {
      "id": "repository",
      "class": "com.example.InMemoryRepository"
    },
    {
      "id": "service",
      "class": "com.example.UserService",
      "constructorArgs": [
        { "ref": "repository" }
      ]
    },
    {
      "id": "app",
      "class": "com.example.Application",
      "constructorArgs": [
        {
          "class": "com.example.GreetingService",
          "constructorArgs": [
            { "ref": "service" }
          ]
        }
      ]
    }
  ]
}
```

Невалидный JSON, циклические ссылки через `ref`, отсутствующий `ref`, несовпадение числа аргументов с конструкторами, несколько подходящих конструкторов с одной арностью — обрабатывайте предсказуемо (`RuntimeException` с понятным сообщением или зафиксированное правило в коде).

## Требования

- `ContextHolder.getApplicationContext("context.json")` возвращает **контекст приложения** с `getBean(String id)` (и при необходимости другими методами для демо/тестов).
- При отсутствии файла — `RuntimeException` с понятным сообщением.
- Поддержаны сценарии: только no-arg; конструктор только из `ref`; конструктор с **вложенными** дескрипторами (`class` + вложенные `constructorArgs`).
- Ошибки загрузки класса, создания экземпляра, разрешения `ref` — `RuntimeException` с указанием проблемного места.
- При дублирующихся `id` в `beans` — на ваше усмотрение, кратко в коде.
- Только стандартная библиотека Java; Java 17+, Maven или Gradle. Spring, Guice, CDI и внешние JSON-библиотеки не использовать.

## Проверка

```java
var context = ContextHolder.getApplicationContext("context.json");
var bean = context.getBean("app");
// проверки: граф зависимостей собран, вложенные бины в конструкторе работают
```

Конфиг в репозитории должен включать минимум один кейс с **`constructorArgs`** и **`ref`**, и минимум один — с **вложенным** объектом (`class` внутри элемента `constructorArgs`).
