# Практика: Чат на Spring Data JPA

**Тег коммита:** `chat-spring-data`
**Баллы:** 4

---

## Описание

Перепишите чат с хранением в PostgreSQL: вместо `JdbcTemplate` и ручного SQL используйте **Spring Data JPA**. Схему БД версионируйте миграциями (Flyway или Liquibase).

## API

| Метод | Путь | Тело запроса | Тело ответа | Статус |
|-------|------|--------------|-------------|--------|
| GET | `/messages` | — | `[{"id","author","text","createdAt"},...]` | 200 |
| POST | `/messages` | `{"author","text"}` | созданный объект | 200 |
| GET | `/messages?author={author}` | — | список сообщений автора | 200 |
| GET | `/users` | — | список пользователей | 200 |
| POST | `/users` | `{"username"}` | созданный объект | 200 |

## Требования

- Данные сообщений и пользователей хранятся в БД; поля соответствуют контракту API (`id`, автор/имя пользователя, текст, время создания для сообщений).
- Время создания сообщения выставляется при сохранении на стороне приложения или БД — без ручной подстановки из клиента в обход правил JPA/DDL.
- Выборка по автору и выборка пользователя по имени выполняются средствами Spring Data без нативного SQL.
- Схема создаётся только миграциями; `spring.jpa.hibernate.ddl-auto=validate`.
- Весь код с `JdbcTemplate`, ручным SQL и `RowMapper` удалён.
- `GET /messages` без фильтра возвращает сообщения в порядке по времени создания (согласуйте направление с ожидаемым UX/API).
- Spring Boot 3.x, Java 17+, PostgreSQL, миграции Flyway или Liquibase. Запрещены: `JdbcTemplate`, `@Query` с нативным SQL, `ddl-auto=create/update`, замена миграций на `schema.sql`.

## Проверка

С **тестовой PostgreSQL** проверьте **эндпоинты** `GET /messages`, `GET /messages?author=…`, `GET /users`, `POST /messages`, `POST /users`: коды ответа и соответствие тел API. После `POST /messages` то же сообщение должно отдаваться через `GET` и/или быть проверяемым в БД в тесте.
